# KanColle Hub 前端使用体验自主优化计划

> 版本：2026-07-10  
> 状态：项目审查完成，后续由 Codex 全程自主实施  
> 范围：仅前端 UI、交互、响应式、可访问性与感知性能；不修改后端业务、数据库结构和 API 语义

## 1. 结论先行

项目不需要推翻现有视觉重做。桌面端的深色“舰队作战台”方向已经成立，1440×900 下作战大厅和舰籍页层级清楚，基础组件、语义色和空状态也已有雏形。当前最值得投入的不是增加装饰，而是依次解决三类问题：

1. **先防误操作**：活动切换会保留旧编辑状态，舰籍视角切换可能把旧数据误标为他人数据，作业卡自动保存没有顺序和失败反馈。
2. **再打通移动端核心任务**：移动导航依赖横滑，锁船页看不到全员矩阵，舰队编辑和选船控件过密，部分游戏扣除资源后却无法触控操作。
3. **最后统一视觉和反馈**：减少过量英文微标签、小字号与 Emoji/Lucide 混用，统一表单、弹层、异步状态、空态、加载态和错误态。

推荐按 4 个可独立验收的实施批次推进，不以人工排期作为停止条件。小游戏先完成“触屏设备不扣资源并明确提示”的安全限制，完整触控适配放在最后按实际收益继续执行。每批都必须独立可上线、可回滚，并通过桌面与移动端验收，不做脱离用户收益的纯重构。

## 3. 优化原则

1. **任务优先于装饰**：每个页面只突出一个主任务，次级管理能力收进二级区域、抽屉或菜单。
2. **活动范围始终可见**：当前活动必须贯穿主导航、页面标题和所有编辑器，不允许静默回到“日常”或另一活动。
3. **移动端不是缩小桌面端**：锁船、选船、舰队编辑、攻略编辑应提供触屏原生流程，不能仅靠横向滚动、Hover 或 HTML5 Drag。
4. **状态必须可解释**：所有修改统一呈现“未保存、保存中、已保存、失败、冲突”；失败不得被静默吞掉。
5. **中文优先、英文克制**：保留品牌和必要术语，减少重复的全大写微标签，让标题、状态和操作一眼可懂。
6. **一致性由基础组件保证**：控件尺寸、圆角、焦点、弹层、表单字段、消息和空态不得由业务页各自定义。
7. **无障碍与移动适配同步完成**：不把键盘、读屏、对比度和减少动态效果留到最后返工。

## 4. 关键问题与优先级

### P0：视觉重构前必须验证并解决的正确性风险

#### P0-1 活动切换可能残留上一活动的编辑状态

这是基于现有 React 状态初始化和同路径 Query 切换得到的高置信风险；批次 A 的第一步应先用交互测试复现并锁定，再实施修复。

证据：

- 活动切换只替换同一路径的 `activityId` Query：`src/components/common/activity-switcher.tsx:14-16,77-80`。
- 锁船的标签、计划、ID 和版本只从初始 Props 初始化一次：`src/components/lock-plan/lock-plan-god-view.tsx:114-155`。
- 作业卡保留正在查看/编辑记录和表单：`src/components/routine/routine-form.tsx:163-174`。
- 攻略保留编辑 ID 与草稿：`src/components/strategy/strategy-editor.tsx:166-178`。
- 三个页面渲染编辑器时均未使用活动 Key：`src/app/(protected)/lock-plan/page.tsx:102-118`、`routine/page.tsx:106-118`、`strategy/page.tsx:67-72`。

处理：

- 短期给三个编辑器设置 `key={scope.scopeKey}`，切活动强制重建本地状态。
- 增加统一 Dirty Guard；有未保存内容时提供“保存并切换 / 放弃并切换 / 取消”。
- 所有提交前校验当前 `activityId` 与草稿所属活动一致。

验收：在活动 A 中打开或修改内容后切到 B，B 不显示 A 的标签、记录、选择和草稿；B 的请求不携带 A 的记录 ID 或标签 ID。

#### P0-2 活动上下文在主导航间断裂

证据：

- Home 支持 `activityId`，但 AppShell 只给作业卡、攻略和锁船透传：`src/app/(protected)/home/page.tsx:24-33`、`src/components/common/app-shell.tsx:26-31,47-51`。
- Dashboard 忽略 URL 范围，固定选择排序后的第一个活动：`src/app/(protected)/dashboard/page.tsx:7-21`。

处理：

- 将 URL 中的 `activityId` 作为唯一前端活动范围来源。
- Home、Dashboard、Routine、Strategy、Lock Plan 的主导航全部保留当前范围。
- 桌面顶部与移动端页面栏展示当前活动名称，并提供统一切换入口。

验收：从任一活动页切换所有一级入口，活动不丢失；浏览器前进/后退可正确恢复范围；舰籍情报和标签与当前活动一致。

#### P0-3 舰籍标签筛选是“看似生效”的死控件

证据：

- UI 维护 `selectedLockTagId` 并显示 `ACTIVE FILTER`：`src/components/ship-data/ship-data-center.tsx:114-116,608-646`。
- `filteredShips` 实际只使用舰种和文本搜索：`src/components/ship-data/ship-data-center.tsx:201-211`。
- 页面只传标签，未传用户的锁船分配：`src/app/(protected)/dashboard/page.tsx:14-20,36-42`。

处理：本批若不补齐前端映射，先隐藏该筛选；若保留，则实现“全部 / 指定标签 / 未贴条”真实结果，并显示筛选前后数量。

验收：任何选中项都与结果集一致；没有数据时明确显示 0 艘，不能仅改变高亮文案。

#### P0-4 舰籍视角切换可能误标数据或回退旧数据

证据：

- 请求成功前先更新 `viewerId`，失败被静默吞掉：`src/components/ship-data/ship-data-center.tsx:119-139`。
- 切回自己时恢复旧 `initialShipData`：`src/components/ship-data/ship-data-center.tsx:120-126`。
- 本次成功保存只更新当前 State：`src/components/ship-data/ship-data-center.tsx:421-436`。
- `ViewerDropdown` 定义在父组件内部，会随父渲染重挂：`src/components/ship-data/ship-data-center.tsx:444-475,509`。

处理：目标数据加载成功后再提交视角；增加 Loading、错误、重试和请求取消；缓存本会话每个用户的最新数据；把下拉组件移出父组件并稳定提供用户列表。

验收：请求失败时仍显示原视角；快速切换时旧请求不能覆盖新选择；保存自己数据后切走再切回仍显示最新结果。

#### P0-5 作业卡分页与自动保存可能给出错误结果

证据：

- records 先按未校正页码 `skip`，查询完成后才算 `safePage`：`src/app/(protected)/routine/page.tsx:23,45-70`。
- 每次舰队变化立即 PATCH，失败被静默吞掉且无顺序控制：`src/components/routine/routine-form.tsx:273-290`。

处理：先 count，再校正或重定向页码后查询；删除末页最后一条后跳上一页。自动保存改为 500–800ms Debounce 的串行最新快照队列，并显示保存状态。

验收：`page=999` 自动到最后一页；删除末页最后一条不出现“第 1 页但空数据”；连续十次编辑最终只保留最后状态；断网时明确显示失败和重试。

### P1：核心移动体验与交互基建

#### P1-1 重做移动一级导航

现状：`AppShell` 在 `lg` 以下使用单行横向导航，390px 下内容宽约 596px、单项高约 34px：`src/components/common/app-shell.tsx:129-149`。

目标：

- 手机使用五项固定底部导航：作战大厅、舰籍、作业卡、锁船、更多；攻略与个人设置从“更多”进入。
- 当前活动切换留在页面顶部，支持横滑但加入边缘渐隐、当前项自动滚入视野。
- 采用 `100dvh`、底部安全区和 `aria-current="page"`。
- 所有移动主操作目标不小于 44×44px。

验收：320px 下所有一级入口最多两次点击可达；直接打开任一深层页面时当前入口可见；页面无文档级横向溢出。

#### P1-2 统一 Dialog、Sheet 与确认交互

现状：通用 Dialog/AlertDialog 没有 `role="dialog"`、标题关联、焦点圈定、焦点恢复或滚动锁；相同 z-index 的全屏容器覆盖遮罩，使遮罩点击不可达：`src/components/ui/dialog.tsx:23-73`、`src/components/ui/alert-dialog.tsx:8-25`。

目标：

- 用一个 Portal Dialog Primitive 统一普通弹层、确认弹层和移动底部 Sheet。
- 支持 Escape、遮罩、明确关闭按钮、Focus Trap、关闭后焦点恢复、背景滚动锁。
- 手机长表单采用全屏 Sheet 或页面级编辑，使用 `100dvh` 和单一滚动容器。

验收：仅键盘可完整操作；Tab 不逃出弹层；关闭后焦点回触发器；320×568 且软键盘弹出时标题、字段、主操作和关闭按钮均可达。

#### P1-3 建立统一控件、表单和异步状态

现状：基础控件无最小高度，表单通常为 14px、局部甚至 12px；登录/注册网络异常可能永久停在提交中，攻略/作业卡提交缺少防重复和失败恢复：`src/components/ui/button.tsx:8-27`、`input.tsx:4-17`、`select.tsx:4-15`、`textarea.tsx:4-15`、`src/components/auth/login-form.tsx:17-37`。

目标：

- 新增 `FormField`、`AsyncButton`、`InlineStatus`、`ToastRegion`、`IconButton`。
- 手机输入字号至少 16px，控件高度 44px；桌面可在 `sm` 以上降到紧凑尺寸。
- 字段统一 `id/htmlFor`、帮助文本、必填标识、错误 `aria-describedby`。
- 所有 Mutation 使用相同状态机：idle、pending、success、error、conflict。
- UI 按 Capability 前置隐藏或禁用无权限操作，并说明原因；当前所有用户都能看到“建立活动档案”，但接口只允许规划者或管理员：`src/components/common/activity-switcher.tsx:93-126`。

验收：iOS 聚焦输入框不触发页面放大；重复点击不重复创建；断网、500 和超时不会永久锁死；错误可被读屏器播报。

#### P1-4 锁船移动端完成“协作全览 + 我的编辑”闭环

现状：移动版只渲染当前用户，全员矩阵在 `md` 以下隐藏；`md` 以上又依赖 HTML5 Drag 和 Hover：`src/components/lock-plan/lock-plan-god-view.tsx:834-984`、`src/components/lock-plan/ship-cell.tsx:56-95`。

目标：

- 移动端增加“我的编辑 / 全员概览 / 冲突”三段视图。
- 全员概览按用户或标签折叠，点击可进入只读详情；冲突摘要可跳到具体成员、舰船和标签。
- “我的编辑”保留显式的追加、上移、下移、移动到标签、移除，不依赖拖拽。
- 平板通过 Pointer 能力而非宽度判断 Hover/Drag；触屏始终显示操作菜单。
- ShipPicker 手机一列或两列，筛选放入 Sheet，列表只有一个滚动容器。

验收：375px 可查看所有有计划成员并定位冲突；当前用户可完成追加、排序、跨标签移动和移除；iPad/Surface 触屏无需鼠标。

#### P1-5 舰籍页改为“导入优先、结果渐进展开”

现状：顶部卡片在 640px 就三列且固定 280px 高，390px 下导入卡形成内嵌滚动，控件约 29–32px 高：`src/components/ship-data/ship-data-center.tsx:477-505`。

目标：

- 手机顶部依次为：同步状态与导入主卡、活动情报摘要、锁船筛选；卡片高度自适应，禁止普通内容区内嵌纵向滚动。
- 导入流程拆成“粘贴数据 → 预览变化 → 确认更新”，主按钮始终靠近当前步骤。
- 舰船/装备手机默认显示精简卡片或关键列；完整数据切换到表格模式，固定关键列并用边缘阴影提示横滑。
- 视角切换展示加载态和只读标记，不允许旧数据带着新用户名显示。

验收：360px 下无需横滑即可看舰名和关键指标；640px 不出现 187px 宽的三列卡片；卡片内部不出现非必要纵向滚动。

#### P1-6 作业卡移动编辑改为分步任务

现状：FleetEditor 所有宽度都双列，舰卡内部固定 30%/70%；列表操作和最多 7 个页码保持单行：`src/components/routine/fleet-editor.tsx:290-330,751-764`、`src/components/routine/routine-form.tsx:401-513`。

目标：

- 手机新建流程分成“基本信息 → 导入/编辑舰队 → 预览并发布”。
- 手机舰队单列、平板双列；舰卡改为上方身份与属性、下方装备列表和显式操作。
- 记录卡标题与操作分两行；次要操作收进菜单。
- 手机分页只保留上一页、当前页/总页数、下一页。
- 页面顶部只显示“新建作业卡”入口，不让空编辑器长期占据列表首屏。

验收：320px 下舰名、等级、装备和操作均完整；有 7 页以上记录时分页不溢出；从打开到发布的主路径在手机上无需横向滚动。

#### P1-7 攻略编辑器从常驻大表单改为按需创作

现状：移动端编辑器和实时预览常驻在列表前方；1024px 起使用 44rem 固定编辑栏，右侧内容区可能只剩约 248px：`src/components/strategy/strategy-editor.tsx:257-329`。

目标：

- 默认以攻略列表为主，“新建攻略”打开全屏/大 Sheet 编辑器。
- 手机使用“编辑 / 预览”Tab，不同时纵向堆叠两块长内容。
- 双栏推迟到 `xl`，使用 `minmax()` 保证帖子正文最小可读宽度。
- 删除增加确认和错误恢复；编辑、上传、插卡、发布统一保存状态。
- 攻略列表的作者、日期和操作在触屏上始终可发现，不依赖 Hover。
- 从列表点“编辑”后自动滚动并聚焦编辑标题，持续显示正在编辑的对象和取消/未保存提示。

验收：1024px 帖子正文宽度可正常阅读；390px 首屏先看到标题、筛选和内容列表；发布失败后草稿仍保留。

#### P1-8 小游戏先阻止资源损失，再补触控

现状：游戏成功扣粮后才打开；Survivor 和 Invaders 只监听键盘并使用固定画布：`src/components/games/game-entry-card.tsx:62-81`、`survivor-game.tsx:10-11,314-344`、`space-invaders.tsx:10-11,266-302`。

目标：

- 热修：检测无可用输入的触屏设备，在开始前禁用并说明“暂仅支持桌面端”，不得先扣资源。
- 完整优化：响应式画布、屏幕方向键/摇杆、射击区、横屏提示、暂停和安全退出。

验收：360×640 竖屏和 844×390 横屏可完成移动、射击、升级与退出；不支持设备不会扣粮后才提示。

### P2：视觉统一、可访问性与感知性能

#### P2-1 收敛设计系统

- 扩展 Token：字号/行高、间距、圆角、阴影、控件高度、Focus Ring、Z-index、Motion、容器宽度。
- 中文标题为主，英文微标签只用于品牌或真正有助识别的技术词。
- 正文与操作说明不低于 14px，手机主体 16px；10/11px 仅用于非关键编号或辅助数据。
- 移除承担操作含义的 Emoji，统一使用 Lucide 或同一图标系统。
- 普通文字对比度目标至少 4.5:1；`slate-600` 不承载必要文本。
- 控制卡片数量和层级，减少“每块内容都包卡片”的视觉噪声。

#### P2-2 补齐语义与键盘等价操作

- 可点击 `span/div/th/tr` 改为真实 Button/Link，表头排序使用 `aria-sort`。
- 当前页使用 `aria-current`，状态消息使用 `aria-live`，确认对话框使用正确角色。
- 拖放必须提供显式的移动、排序和移除替代操作。
- 200% 缩放不丢内容；仅用 Tab、Enter、Space 和方向键可完成核心流程。

#### P2-3 补齐路由加载、错误和空状态

- 为受保护路由添加 `loading.tsx`、`error.tsx` 和可重试边界。
- 空态区分：尚未导入、无匹配、解析失败、无权限、网络离线、活动未建立。
- Skeleton 只模拟稳定布局，避免加载后大幅跳动。
- 登录支持安全 `next` 回跳；分享链接登录后回原活动页面。
- 头像和背景更新采用局部状态刷新；不得用整页 Reload 造成闪白、滚动位置和未保存上下文丢失。

#### P2-4 优化图片、主数据与大型组件

- 修复 8 处 `<img>` 警告；头像等固定尺寸图片提供明确尺寸、懒加载和解码策略。
- 用户背景移动端停用 `background-attachment: fixed`，提供低分辨率或弱化版本。
- 主数据按版本缓存，避免二次访问重复下载完整约 2MB 数据；需要时再加载，并展示明确状态。
- 长表格使用虚拟化或分页；ReactMarkdown 与小游戏按需加载。
- 攻略实时预览使用 Deferred/节流，长攻略列表分页或延迟渲染，避免每次键击同步重算整页 Markdown。
- 拆分 `ShipDataCenter`、`LockPlanGodView`、`FleetEditor`，但每次拆分必须绑定一个可验证的交互或性能收益。

## 7. 统一验收标准

### 7.1 视口与设备矩阵

- 320×568、360×800、390×844。
- 640×960、768×1024。
- 1024×768、1280×800、1440×900。
- iOS Safari、Android Chrome、桌面 Chrome/Edge；至少一次触屏平板实机或等价测试。

### 7.2 响应式

- `document.documentElement.scrollWidth <= clientWidth`；明确设计的局部数据表横滑除外。
- 320px 下无文字覆盖、操作裁切、不可达按钮或嵌套页面级滚动。
- 手机主操作至少 44×44px，任何可操作目标不低于 24×24px。
- 软键盘、地址栏收放、刘海安全区和横屏不会遮挡主操作。

### 7.3 交互正确性

- 活动切换不残留旧活动状态；前进/后退恢复正确。
- 所有 Mutation 防重复；失败保留输入并提供重试。
- 离开未保存表单或切活动前有明确保护。
- 删除、扣资源和覆盖类操作必须在执行前确认或满足可撤销条件。
- 请求竞态不能让旧响应覆盖新选择。

### 7.4 可访问性

- axe 无 Serious/Critical。
- 普通文本对比度目标至少 4.5:1，大字至少 3:1。
- 200% 缩放不丢内容。
- 仅键盘可完成登录、导航、筛选、弹层、选船、排序、编辑与保存。
- 焦点可见；弹层焦点不逃逸且关闭后回到触发器。
- 开启 `prefers-reduced-motion` 时取消非必要动画。

### 7.5 感知性能与稳定性

- 路由切换有稳定 Skeleton 或明确 Loading，不闪现误导性空数据。
- 头像等固定尺寸图片无明显 CLS，不下载不必要的原始大图。
- 主数据二次访问不重复下载未变化的完整数据。
- 长列表滚动和输入筛选保持流畅；无输入导致的无关组件重挂或重复请求。

### 7.6 工程门槛

- `npm test`、`npm run lint`、`npm run build` 全通过。
- 新增交互必须有组件或 E2E 回归；至少覆盖活动切换、舰籍视角、分页、自动保存、Dialog、移动锁船。
- 每个实施批次保持工作树可审查，提交按用户结果拆分，不混入后端业务改动。

## 8. 自主执行方式

每个优化项按以下闭环执行：

1. 记录当前行为、受影响页面和最小复现。
2. 先补可失败的回归测试或明确手工验收脚本。
3. 优先修复公共基础能力，再改业务页面；避免多个页面重复实现。
4. 在 390×844 和 1440×900 完成视觉巡检，再补 320/768/1024 边界视口。
5. 验证键盘、触屏、断网、慢请求、重复点击和浏览器前进/后退。
6. 运行 Test、Lint、Build；记录仍存在的警告。
7. 单项验收通过后再进入下一项，不在同一批次混入无关重构。

默认由 Codex 直接采用以下方案，无需中途等待产品选择：

- 锁船移动全览默认按用户组织，同时提供按标签切换。
- 小游戏先限制不支持的触屏设备且不扣资源，再在批次 D 评估并继续完整触控适配。
- 移动导航、页面断点、组件拆分和视觉收敛均按本计划默认方案执行。

仅当实现必须新增后端数据、改变 API 合约或扩展业务规则时暂停，并说明缺口与最小后端需求；其余前端判断由 Codex 自主完成。
